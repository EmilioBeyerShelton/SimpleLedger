# SimpleLedger — React port architecture

This document explains how `portToReact/` is put together and why, for
whoever (human or Claude) touches it next. It's a from-scratch React port
of the original Preact/htm, no-build-step app at the repo root — same
features, same data shape, three build targets instead of one.

## Goals that shaped every decision

1. **One codebase, three shells.** Web (static site / PWA), iOS
   (Capacitor), macOS (Electron). The React component tree and the
   Zustand store are 100% shared. Only the persistence layer branches per
   platform.
2. **Feature parity with the original**, not a redesign. Every view, every
   piece of business logic (double-entry accounting, budget splitting,
   sankey/pie charts, the legacy-data migration) is a direct port. Look at
   the top-of-file comment in each source file — it names the original
   `js/*.js` file it replaces.
3. **Mobile-first, works everywhere.** Bottom tab bar on narrow viewports,
   the same bar becomes a top bar on wide/desktop layouts (`md:` Tailwind
   variants in `BottomNav`/`App`). `env(safe-area-inset-*)` is respected
   for iOS notches/home indicators.

## Directory layout

```
    lib/

      persistence/
        types.ts             PersistenceAdapter contract every platform implements
        normalize.ts          ported js/store.js `normalize()`/`defaultData()` — legacy data migration
        web.ts                Web adapter: localStorage + File System Access API
        capacitor.ts           iOS adapter: Preferences + Filesystem + Share
        electron.ts             macOS adapter: IPC to the main process (electron/main.cjs)
        electron.d.ts            types for window.electronLedger
        index.ts                 platform detection + adapter factory
    store/
      useLedgerStore.ts      Zustand store — all state + mutations, talks only to PersistenceAdapter
    components/

      layout/                TopBar, BottomNav

```

## State management: Zustand

`src/store/useLedgerStore.ts` is the single source of truth. It holds the
whole `LedgerData` object (accounts, transactions, groups,
groupTransactions, settings) plus `fileStatus` and a reference to the
active `PersistenceAdapter`. Every mutation goes through `mutate()`, which
mirrors the original `store.js`'s `update()`:

1. Compute the next state.
2. `set()` it (React re-renders).
3. Fire-and-forget `persistLocal()` (always-on store) and
   `writeLinkedFile()` (only if a file is linked) through the adapter.

Pages select just the slice/actions they need
(`useLedgerStore(s => s.data)`), so unrelated state changes don't cause
extra re-renders. There's deliberately no derived-state caching beyond
`useMemo` inside components — the dataset is small (personal finance data,
not a firehose) so this stays simple rather than fast-but-complicated.

## Persistence layer

This is the part that's genuinely different from the original app, and
the reason the project exists as `portToReact/` rather than just replacing
the old files in place.

Every platform implements the same `PersistenceAdapter` interface
(`src/lib/persistence/types.ts`):

- `loadInitial()` — read the always-on store, reconcile with a linked file
  if one was remembered, migrate legacy shapes via `normalize()`.
- `persistLocal(data)` — write to the always-on store. Called on every
  mutation.
- `writeLinkedFile(data)` — mirror to the linked file, if any.
- `connectExisting()` / `connectNew()` / `reconnect()` / `disconnect()` —
  manage the optional linked-file relationship.
- `downloadBackup(data)` / `uploadBackup(file)` — manual JSON
  export/import, independent of any linked file.

### Web (`web.ts`)

Unchanged behavior from the original `js/store.js`: `localStorage` is the
always-on store; an optional **File System Access API** handle
(Chrome/Edge only — feature-detected via `showOpenFilePicker`) is the
linked file, with the handle itself persisted in IndexedDB since it isn't
JSON-serializable. Firefox/Safari get `supported: false` and just use
localStorage, same graceful degradation as before.

### iOS (`capacitor.ts`)

iOS has no equivalent of a persistent external-file handle, so the design
had to change shape rather than being a literal port:

- **Always-on store**: `@capacitor/preferences` (backed by
  `NSUserDefaults`) — analogous to localStorage, survives app updates and
  device backups.
- **"Linked file"**: instead of linking an arbitrary external file, we
  mirror every save to `Documents/ledger-data.json` inside the app
  sandbox. With `UIFileSharingEnabled` set in `Info.plist` (see the iOS
  setup section below), that file shows up under **Files app → On My
  iPhone/iPad → SimpleLedger**, so the user can inspect, AirDrop, or back
  it up through Files like any other document — the closest iOS
  equivalent to "a file on disk."
- **Backup**: `downloadBackup` writes a timestamped snapshot to the cache
  directory and opens the native share sheet (`@capacitor/share`) so the
  user can save it to Files, AirDrop it, etc.
- **Restore**: a plain `<input type="file">`, which iOS's WKWebView routes
  to the native document picker — no extra plugin needed.

### macOS (`electron.ts`)

Electron's renderer process is sandboxed (`contextIsolation: true`,
`nodeIntegration: false`), so it never touches Node's `fs` directly.
`electron/preload.cjs` exposes a narrow `window.electronLedger` bridge;
`electron/main.cjs` implements the other end over `ipcMain.handle`:

- **Always-on store**: a JSON file in `app.getPath('userData')`
  (`~/Library/Application Support/SimpleLedger/ledger-data.json`) — the
  macOS analogue of localStorage.
- **Linked file**: a second, user-chosen path (native save/open dialogs
  via `dialog.showOpenDialog`/`showSaveDialog`), mirrored on every save —
  same UX as the web adapter's File System Access API handle. The chosen
  path is remembered in the renderer's `localStorage` (which Electron
  persists fine across launches) rather than in the main process, since
  it's just a string.

### Adapter selection (`index.ts`)

```
window.electronLedger present        -> ElectronPersistenceAdapter (macos)
Capacitor.isNativePlatform() && ios   -> CapacitorPersistenceAdapter (ios)
otherwise                             -> WebPersistenceAdapter (web)
```

Selected once, cached, and injected into the store at `init()`. Nothing
else in the app branches on platform except `SettingsPage`, which adjusts
button labels/copy (e.g. "Download JSON" vs. "Share backup").

## Legacy data migration

`src/lib/persistence/normalize.ts` is a line-for-line TypeScript port of
`js/store.js`'s `normalize()`/`defaultData()`. It handles:

- Old numeric account ids → new dotted-path string ids.
- Old inline `transaction.groupId`/`transaction.splits` → the
  `groupTransactions` join table.

Every adapter's `loadInitial()`, `connectExisting()`, `reconnect()`, and
`uploadBackup()` path runs data through `normalize()`, so any export from
the original vanilla-JS app — or an even older export — loads correctly
on any platform.

## UI: shadcn/ui + Tailwind

`src/components/ui/*` are hand-vendored shadcn/ui primitives (Button,
Input, Dialog, Select, Checkbox, Tabs, Slider, Card, Badge, Label) built
on Radix primitives + `class-variance-authority` + `tailwind-merge`, the
same pattern the shadcn CLI generates. They're committed to the repo
(shadcn is a copy-paste library, not an npm dependency) so they can be
freely edited — see the project rules in `.claude/CLAUDE.md` for the
convention this implies.

Theme tokens live in `src/styles/globals.css` as CSS variables
(`--primary`, `--background`, etc.), matching the original app's warm
green/cream palette from `index.html`'s inline `<style>`.

## Routing

`react-router-dom`'s `createHashRouter` (not `BrowserRouter`): both
Electron and Capacitor load `index.html` as a static file with no server
to rewrite history-API paths, and hash routing (`#/report`, `#/accounts`,
...) works identically across web, macOS, and iOS with zero server
config. `App.tsx` is the router's layout route — it boots the store once,
shows a loading state until the first `PersistenceAdapter.loadInitial()`
resolves, then renders `<Outlet />` for the active tab.

## Build targets

| Target | Command | Output |
|---|---|---|
| Web | `npm run build` | `dist/` — deploy as a static site |
| macOS | `npm run electron:build` | `.dmg` via electron-builder |
| iOS | `npm run cap:ios` | opens the generated Xcode project |

`vite.config.ts` uses `base: './'` (relative asset paths) specifically so
the same `dist/` bundle loads correctly under `file://` in Electron and
in Capacitor's WKWebView, not just from an HTTP server.

## What's intentionally not ported 1:1

- The original's `?v=N` cache-busting query strings on script tags — Vite
  handles content-hashed filenames itself.
- Preact/htm's classic-script, no-CORS-under-`file://` trick — Vite
  bundles everything into hashed ES modules and each shell (web server,
  Electron, Capacitor WKWebView) serves/loads them in a way that doesn't
  hit the `file://` + `type="module"` CORS wall the original had to work
  around.
